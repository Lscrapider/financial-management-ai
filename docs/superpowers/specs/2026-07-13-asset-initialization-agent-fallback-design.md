# 新增标的初始化与 Agent 数据兜底设计

## 背景

新增股票目前只保存配置和实时快照，并同步分时；新增可转债会保存基础资料、正股配置和部分行情数据，但不会补齐正股基本面与可转债份额变动。Agent 的基本面工具只读本地库，因此刚新增的标的可能返回空数据。

Report 在提交生成时有一套按需拉源并落库的逻辑，造成“报告能查到、Agent 查不到”的路径不一致。可转债 Report 对日度估值也只读库，不能补齐从未成功写入的 `cb_daily` 数据。

## 目标

- 新增标的完成必要校验和核心数据落库后立即返回，不等待慢数据源。
- 后台初始化股票行业、估值、财务指标、分红，以及可转债行情、日度估值和份额变动。
- Agent 首次查询发现数据缺失或过期时，同步调用同一套确保逻辑并返回可用数据。
- Report、后台初始化、Agent 复用同一套数据新鲜度、数量上限和持久化规则。
- 修正可转债日度估值“空列表仍报告同步成功”的状态语义。

## 非目标

- 不新增数据库表、初始化任务状态页、重试队列或前端进度轮询。
- 不改变现有行情、基本面数据源及其默认配置。
- 不让 Python Agent 直接访问数据源或数据库。
- 不改变 Agent action、Python 工具签名和既有 `data` 字段结构。

## 方案选择

采用“后台初始化 + Agent 首查兜底”的方案。

纯 Agent 兜底虽然改动更小，但从未发起过对话的新标的会长期缺少慢数据。持久化任务状态和重试机制可靠性更高，但会扩展迁移、接口和前端，超出当前问题范围。

## 同步边界

以下校验必须同步执行，否则无法防止错误代码或名称入库：

- 股票：查询实时快照并校验股票代码、名称，随后保存 `stock_config` 与快照。
- 可转债：查询基础资料并校验债券代码、名称，随后保存 `bond_config` 与基础资料。

保存成功后立即返回，并将下列工作投递到后台：

- 股票：分时同步，以及行业、估值历史、财务指标、分红历史初始化。
- 可转债：报价、分时、日/周/月 K 线、日度估值、份额/转股变动初始化。
- 可转债自动新增的正股走股票新增逻辑，因此同样会进入股票初始化。

背景执行沿用项目现有 `CompletableFuture.runAsync` 模式，不引入新的线程池配置或同义阈值。当前部署为单实例；同一标的的后台刷新和 Agent 首查由共享服务合并，避免重复访问外部数据源。

## 共享服务设计

在 `finance-service` 增加无 AI DTO 依赖的 `AssetDataEnsureService`，作为唯一的数据补齐入口。

### 股票确保逻辑

服务复用并下沉现有 Report 的规则：

1. 按既有时效检查行业信息、估值历史、财务指标和分红历史。
2. 缺失或过期时调用既有 `StockFundamentalProvider`，并通过对应 Manage 保存。
3. 行业接口未返回数据时，复用既有“估值历史中的板块名称回填行业”逻辑。

现有的估值数量上限、财务/分红数量上限和刷新天数将从 AI 层迁移到该服务，保持数值及语义不变，不重新定义业务常量。

### 可转债确保逻辑

服务复用既有 `ConvertibleBondDataProvider`、Manage 和 `bond.sync.convertible-daily-limit` 配置：

1. 基础资料、份额/转股变动按既有刷新时效补齐。
2. 日度估值在库中为空时拉取历史数据、保存后重新确认至少存在一条对应债券记录。
3. Provider 缺失、返回空列表或保存后仍无记录时，明确标记为未获得数据，而不是报告同步成功。

## 调用关系

```text
新增股票/可转债
  -> 校验并保存配置、快照或基础资料
  -> AssetDataInitializationService（后台）
      -> 行情任务
      -> AssetDataEnsureService

Agent Handler（首次发现缺失或过期）
  -> AssetDataEnsureService（同步等待）
  -> 重查本地库并组装原有响应

Report
  -> SceneAssetDataEnsureService
  -> AssetDataEnsureService
  -> 只负责 AI DTO 转换和快照覆盖
```

`finance-ai` 已依赖 `finance-service`，因此共享服务必须放在 `finance-service`；不得让新增业务依赖 AI 模块的 `SceneAssetDataEnsureService`，以免形成循环依赖。

## 并发与失败处理

- 共享服务按“标的类型 + 标的代码”串行进行中的刷新。后台任务正在刷新时，Agent 首查先等待本轮刷新结束，再复查并仅补齐自己仍缺少的数据；若后台尚未开始，Agent 自己成为刷新调用方。这样不同补齐范围不会误用彼此的结果，也不会并发重复访问同一数据源。
- 后台任务捕获异常并记录中文日志，不影响新增接口成功返回。
- Agent 的刷新异常不覆盖已有本地数据。Handler 仍返回原有数据结构，并在实际触发兜底时于 `metadata.dataRefresh` 追加刷新状态；对外仅返回稳定的 `refresh_failed` 代码，完整异常只写服务端日志。
- `metadata.dataRefresh` 仅追加字段，包含是否尝试、状态、已请求分区和失败原因（如有）；兼容 Python 工具、模型和前端的现有消费方式。

## 接口与前端兼容

新增结果 VO 增加 `initializationScheduled=true`。

- 已同步落库的快照或基础资料仍按事实返回成功。
- 尚在后台处理的 `trendSynced`、`marketDataSynced`、`dailyValuationSynced` 和 `shareSynced` 不再伪装为已完成。
- 系统配置页面使用 `initializationScheduled` 显示“已加入后台初始化”，不把未完成状态提示为同步失败。

不新增状态查询接口；本次不提供后台初始化进度或失败重试页面。

## Report 改造

`SceneAssetDataEnsureService` 保留原有接口和 DTO 行为，但改为委托 `AssetDataEnsureService` 完成外部拉取与持久化。股票报告维持现有结果；可转债报告在日度估值缺失时也能够补齐该数据，再进行现有快照覆盖。

## 验收标准

- 新增股票或可转债在完成必要校验后不等待慢数据初始化即可返回，并明确标记已排队。
- 后台初始化完成后，股票基本面工具和可转债上下文工具均能读取对应数据。
- 在后台尚未完成时，Agent 首查会触发同一套补齐逻辑，不再仅返回空集合。
- 可转债日度估值返回空时，新增和补齐状态不会报告成功。
- Report 与 Agent 不再维护相互独立的外部拉取规则。
- 不影响已有 action、Python 工具参数、数据库结构和行情数据源配置。

## 验证策略

用户已明确授权本次新增聚焦单元测试。实施按测试先行方式覆盖共享确保服务、后台初始化状态和 Agent 缺失数据兜底；同时执行已有相关测试、Maven 编译和静态检查。外部数据源真实调用仍通过日志和一次受控手工新增流程验证。

## 规格自检

- 规格内容完整。
- 共享服务位于正确的依赖方向，未引入 `finance-service -> finance-ai`。
- 同步校验、后台初始化和 Agent 兜底边界明确。
- 未新增默认业务阈值，复用现有配置与常量。
- 范围未扩展到任务持久化、前端进度和重试系统。
